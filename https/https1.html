HTTPS

1. HTTPS

HTTPS는 HTTP 요청을 SSL 혹은 TLS라는 알고리즘을 이용해 HTTP 통신을 하는 과정에서 내용을 암호화하여 데이터를 전송하는 방법이다.

인증에서 HTTPS 프로토콜을 사용해야만 하는 이유는 HTTP보다 상대적으로 안전한 방법이고 데이터 제공자의 신원을 보장받을 수 있기 때문이다.

HTTPS 프로토콜의 특징 중 하나인 암호화는 암호화된 데이터를 주고받기 때문에 중간에 인터넷 요청이 탈취되더라도 중요한 데이터가 유출될 가능성이 HTTP보다 현저히 적어지게 된다.

HTTPS 프로토콜의 또 다른 특징 중 하나는 브라우저가 응답과 함께 전달된 인증서 정보를 확인할 수 있다는 점이다.

2. SSL & TLS

SSL(Secure Sockets Layer)은 암호화 기반 인터넷 보안 프로토콜이다.
웹사이트와 브라우저 또는 두 서버 사이에 전송된 데이터를 암호화하여 인터넷 연결을 보안을 유지하는 표준 기술이다.
이는 해커가 개인 정보 및 금융 정보를 포함한 전송되는 모든 정보를 열람하거나 훔치는 것을 방지한다.

TLS(Transport Layer Security)는 가장 최신 기술로 더 강력한 버전의 SSL로 같은 의미이다.

SSL 인증서

SSL 인증서는 클라이언트와 서버간의 통신을 제3자가 보증해주는 전자화된 문서이다.

클라이언트가 서버에 접속한 직후에 서버는 클라이언트에게 인증서 정보를 전달하게된다.
클라이언트는 이 인증서 정보가 신뢰할 수 있는 것인지를 검증 한 후에 다음 절차를 수행하게된다.

브라우저는 인증서에서 해당 인증서를 발급한 CA(공인 인증서 발급 기관) 정보를 확인하고 인증된 CA(공인 인증서 발급 기관)가 발급한 인증서가 아니라면 화면에 경고창을 띄워 서버와 연결이 안전하지 않다는 화면을 보여준다.

SSL 인증서의 역할은 클라이언트가 접속한 서버가 신뢰할 수 있는 서버임을 보장하며 SSL 통신에 사용할 공개키를 클라이언트에게 제공한다.

CA

인증서의 역할중 하나가 클라이언트가 접속한 서버가 신뢰할 수 있는 서버임을 보장한다고 하였는데 이 역할을 하는 민간기업들을 CA(Certificate authority)라고 부른다.
CA는 아무 기업이나 할 수 있는 것이 아니고 신뢰성이 엄격하게 공인된 기업들만 참여할 수 있다.

SSL을 통해서 암호화된 통신을 제공하려는 서비스는 공인된 CA가 제공하는 인증서를 사용해야한다.

SSL 암호화의 종류

암호화란 일련의 정보를 임의의 방식을 사용하여 다른 형태로 변환하여 해당 방식에 대한 정보를 소유한 사람을 제외하고 이해할 수 없도록 알고리즘을 이용해 정보를 관리하는 과정이다.

SSL의 핵심은 암호화라고 할 수 있으며 보안과 성능상의 이유로 대칭키, 공개키 두가지 암호화 기법을 혼용해서 사용하고 있다.

대칭키 암호화 방식

대칭키는 동일한 키로 암호화와 복호화를 같이 할 수 있는 방식의 암호화 기법을 의미한다.

대칭키 암호화 방식은 암호를 주고 받는 사람들 사이에 대칭키를 전달하는 것이 어렵고 대칭키가 유출되면 키를 획득한 공격자는 암호의 내용을 복호화 할 수 있기 때문에 암호가 무용지물이 되는 단점이 있다.
이를 보완하기 위해서 나온 방법이 공개키 암호화 방식이다.

공개키 암호화 방식

공개키 방식은 두개의 키를 갖게 되는데 암호화와 복호화에 서로 다른 키를 사용하는 방식의 암호화 기법을 말한다.
즉, A키로 암호화를 하면 B키로 복호화 할 수 있고 B키로 암호화하면 A키로 복호화 할 수 있는 방식이다.

두개의 키 중 하나를 비공개키 나머지 하나를 공개키로 지정하고 비공개키는 자신만이 가지고 있고 공개키는 타인에게 제공한다.
공개키를 제공 받은 타인은 공개키를 이용해서 정보를 암호화하고 암호화된 정보를 비공개키를 가지고 있는 사람에게 전송한다.
비공개키의 소유자는 이 키를 이용해서 암호화된 정보를 복호화 한다.

공캐키 암호화 방식에서 공개키가 유출된다고 해도 비공개키를 모르면 정보를 복호화 할 수 없기 때문에 대칭키 암호화 방식보다 안전하다.

SSL 통신과정

컴퓨터와 컴퓨터가 네트워크를 통해서 통신을 할때 핸드쉐이크 -> 세션 -> 세션종료의 과정을 거친다.

Handshake(핸드쉐이크)

실제 데이터를 주고 받기 전에 클라이언트와 서버는 Handshake과정을 거친다.
이 과정을 통해서 서로 상대방이 존재하는지 상대방과 데이터를 주고 받기 위해서는 어떤 방법을 사용해야 하는지를 파악한다.

SSL 방식을 이용해서 통신을 하는 브라우저와 서버 역시 Handshake과정을 거치는데 이 때 SSL 인증서를 주고 받게된다.

(1) 클라이언트가 서버에 암호화된 정보를 전송한다.

(2) 서버가 클라이언트에 암호화 방식과 인증서를 포함한 정보를 전송한다.

(3) 클라이언트는 서버의 인증서가 CA에 의해서 발급된 것인지 확인하며 확인이 완료되면 서버에서 넘어온 정보를 확인하고 비밀키를 생성한다.

(4) 클라이언트는 서버에 비밀키 정보를 전송해 서버와 클라이언트 양쪽 모두 비밀키를 공유한다.

(5) 클라이언트는 서버에 클라이언트의 인증서를 보낸다.

(6) 서버가 필요한 경우 클라이언트의 인증서를 확인한다.

(7) 클라이언트는 Handshake 단계의 종료를 서버에게 알린다.

(8) 서버는 Handshake 단계의 종료를 클라이언트에게 알린다.

(9) 서버와 클라이언트는 암호화된 메세지를 교환한다.

세션

세션은 실제로 서버와 클라이언트가 데이터를 주고 받는 단계이다.
이 단계에서 핵심은 정보를 상대방에게 전송하기 전에 session key 값을 이용해서 대칭키 방식으로 암호화 한다는 점이다.

세션종료

데이터의 전송이 끝나면 SSL 통신이 끝났음을 서로에게 알려준다.
이 때 통신에서 사용한 대칭키인 session key를 폐기한다.
